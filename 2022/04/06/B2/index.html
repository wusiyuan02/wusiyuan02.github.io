hexo + githubPages 生成博客的原理

Hexo+ GithubPages 搭建的博客本质是一个web网站。web网站有静态和动态区分，静态网站是指用户和web页面不能做一些交互，动态网站允许用户和网站做一些交互。博客网站类似于一个静态网站。也就是说输入博客的url：wusiyuan02.github.io，就会进入正常的输入url解析的过程。

github Pages

github 自己用的很少，所以对github的了解也就很少，github 在我看来可能就是一个代码管理平台？马上就得熟悉一下github这东西。先了解一下pages服务吧~

GitHub Pages是GitHub公司提供的免费静态网站托管服务。GitHub Pages严格意义上来说并不是一个服务器，只是可以提供类似服务器功能的一种服务。当我们把HTML等资源文件存放到GitHub指定的位置时，也就是一个GitHub Pages仓库下，GitHub Pages服务会对这些文件进行处理并把它展示为一个网站。所以说可以将GitHub Pages提供的功能替代Web服务器的功能。我们只需要把编写好的HTML等资源文件存到GitHub指定的位置，那么就实现了类似的Web服务器的功能，可以响应请求，并把相应的资源文件发送给客户端。

hexo 使用markdown 解析文件，根据主题生成静态网页。

这是hexo文件的架构，很显然，我们写的.md文件是存放在source里面的。

每次部署的流程有

1、 hexo g 生成静态文件。将我们的md文件和theme主题中的界面模板结合生成静态文件。hexo g会遍历主题文件中的source 目录，获取静态资源，然后建立索引，根据索引生成由html、css、js、img建立的纯静态文件然后放到public文件夹中。

2、hexo d 部署文件。根据_config.yml 配置的git仓库将public文件通过git方法push到github上。

3、然后就可以根据pages 服务呈现页面了。

移动端的适配方案

下面来补充一下移动端的适配方案！

移动端的适配目的是为了让不同屏幕大小的终端设备拥有一致的UI页面。和这方面息息相关的就是viewport，meta标签中的name属性的一个值。

viewport

viewport叫做视口，也可以叫做视窗。在vscode快捷生成html模板的时候都会有一句：

就是说，移动端为了让页面能更好的展示内容，视口的宽度默认为设备的宽度。视窗有三个概念：布局视窗，视觉视窗，理想视窗。

  • 布局视窗（layout viewport）：浏览器中css布局的区域，如果不加上面那一HTML语句，不同设备不同浏览器的默认值不同，大多数都默认为980px。此时浏览器默认的把页面缩小了。此时的缩放值为理想视窗 / 布局视窗。

    • 网页中的所有内容，可以通过document.documentElement.clientWidth来获取。

  • 视觉视窗（visual viewport）：设备显示网页的窗口，可以拖动或者放大缩小网页。

    • 可以通过window.innerWidth来获取。

  • 理想视窗（ideal viewport）：当前设备展示给用户内容的页面的窗口。

我们想要达成的适配的效果，是再没有特殊布局要求下，让布局视窗和理想视窗达成一致。

再说一下content里面的属性：

  1. width: 定义视口的宽度，取值可以为device-width或者正整数。

  2. height：定义视口的高度，取值可以为device-height或者正整数。

  3. initial-scale：初始化缩放值。取值为小数或者整数。

  4. maxinum-scale：最大缩放值，整数或者小数。

  5. mininum-scale：最小缩放值，整数或者小数。

  6. user-scalable：定义用户是否可以缩放，yes / no。

缩放值 = 理想视窗 / 视觉视窗

当我们设置缩放比为0.5时，在iphone6/7/8中，通过window.innerWidth 得750，即视觉视窗就为750px，此时理想视窗为375px，而理想视窗是不变化的，理想视窗和设备有关，所以公式应该改成下面这样：

视觉视窗 = 理想视窗 / 缩放值

移动端的适配方案

rem

以HTML的font-size为参考，修改HTML元素的字体大小可以成比例的调整以rem为单位的属性。

在移动端浏览器 布局中，rem 布局是解决移动端适配的主流方案。

配合媒体查询检测不同的屏幕尺寸设置不同的HTML元素的font-size属性。

使用vscode写rem时，提供了一个postcss pxtorem 可以将px转换成rem。

em

以父元素的font-size为参考，主要缺点是父元素的font-size难以确定。

vw/vh

vw和vh是针对于理想视口的。将理想视口的width和height分成100份，1vw/vh 就是理想视口的百分之一。

百分比布局

给宽和高设一百分比，缺点是元素嵌套比较深的时候会出现问题。而且一些元素的百分比参照不同，增加开发的难度。

常见的百分比参考

  1. 子元素的width、margin、padding参考父元素的width
  2. 子元素的height参考父元素的height
  3. 子元素的transform、border-radius参考自身。
